**回答（Answer）：**

你的代码在尝试通过反射获取 `Student` 类的 `name` 字段时抛出了 `java.lang.NoSuchFieldException`。这是因为：

- `Class.getField(String name)` **只能获取** `public` 修饰的字段（包括从父类继承的 public 字段）。
- 如果 `Student` 类中的 `name` 字段是 `private`、`protected` 或默认（包可见）的，该方法就会抛出 `NoSuchFieldException`。

从你输出 `Student{name = null, age = 0}` 来看，`Student` 类很可能将字段声明为 `private`，并重写了 `toString()`，所以即便字段私有，`toString()` 仍能访问它们。

---

### 解决方案

使用 **`getDeclaredField`** 替代 `getField`，它可以获取当前类中**所有访问级别**的字段（包括 private），然后通过 `setAccessible(true)` 绕过访问控制。

修改你的代码：

```java
// 原错误代码
Field field = clazz.getField("name");
field.setAccessible(true);
System.out.println(field.get(st));

// 修改为：
Field field = clazz.getDeclaredField("name");  // 获取私有字段
field.setAccessible(true);                     // 设置可访问
System.out.println(field.get(st));
```

---

### 补充说明

1. **`getDeclaredField` 与 `getField` 的区别**：
   - `getField`：只能获取 `public` 字段，且包括父类的 public 字段。
   - `getDeclaredField`：获取本类中**任意访问修饰符**的字段，但不包括父类的字段（除非显式在父类上调用）。
2. **访问权限**：
   - 对于私有字段，必须调用 `setAccessible(true)`，否则后续的 `get` 或 `set` 操作会抛出 `IllegalAccessException`。
3. **你的代码其他部分**：
   - `clazz.newInstance()` 在 Java 9+ 中已标记为过时，建议改用 `clazz.getDeclaredConstructor().newInstance()`，但当前不影响功能。

### 深入底层，看看代理对象到底是什么

为了让你看得更透彻，我们来看看 JDK 动态代理在运行时到底干了什么。

当你调用 `Proxy.newProxyInstance(ClassLoader, new Class[]{Star.class}, handler)` 时，JVM 会在内存中**动态生成**一个新的类（通常类名类似 `$Proxy0`）。这个新类的源码结构大致如下：

```java
// 这是 JVM 在内存中动态生成的类，并非你手动编写的
public final class $Proxy0 extends Proxy implements Star {  // 👈 关键点：实现了 Star 接口！

    // 构造方法，传入 InvocationHandler
    public $Proxy0(InvocationHandler h) {
        super(h);
    }

    // 实现 Star 接口的 sing 方法
    @Override
    public final String sing(String name) {
        try {
            // 所有逻辑都转发给 InvocationHandler 的 invoke 方法
            return (String) super.h.invoke(this, method_sing, new Object[]{name});
        } catch (RuntimeException | Error e) {
            throw e;
        } catch (Throwable e) {
            throw new UndeclaredThrowableException(e);
        }
    }

    // 实现 Star 接口的 dance 方法
    @Override
    public final void dance() {
        try {
            super.h.invoke(this, method_dance, null);
        } catch (RuntimeException | Error e) {
            throw e;
        } catch (Throwable e) {
            throw new UndeclaredThrowableException(e);
        }
    }
}
```

1. **它算是对 `Star` 的实现吗？**

2. **为什么可以用 `Star` 来接收？**

   - `newProxyInstance` 返回的是 `$Proxy0` 类型的对象。
   - `$Proxy0` 实现了 `Star` 接口。
   - 子类对象赋值给父接口引用（向上转型），这是完全合法的多态用法。所以编译器不会报错，运行时也不会类型转换异常。

**1. 代理对象有类吗？可以获取 Class 吗？**
**绝对有，而且完全可以获取。**
只要是 Java 对象，必定源于某个类。你可以在 `test01` 中加上这行代码验证：

```java
System.out.println("代理对象的类名：" + starProxy.getClass().getName());
System.out.println("代理对象的父类：" + starProxy.getClass().getSuperclass().getName());
```

控制台会输出：

```text
代理对象的类名：com.sun.proxy.$Proxy0
代理对象的父类：java.lang.reflect.Proxy
```

**2. 它与普通的类（如 `BigStar`）有什么核心差别？**

| 维度 | 普通类（如 `BigStar`） | 动态代理类（如 `$Proxy0`） |
| --- | --- | --- |
| **生成时机** | **编译时**（写死在 `.java` 文件里） | **运行时**（JVM 在内存中动态生成字节码） |
| **继承体系** | 继承 `Object`，可实现任意接口 | **强制继承 `java.lang.reflect.Proxy`**（这是 JVM 的硬性规定） |
| **类修饰符** | 可以是 `abstract`、`final` 或普通类 | 一定是 **`public final`**，不能被继承 |
| **方法实现** | 包含具体的业务逻辑代码 | 所有方法体内**只有一行核心逻辑**：`super.h.invoke(...)`（全部转发给 `InvocationHandler`） |
| **构造方法** | 可以有无参、有参构造 | **只有一个构造方法**，且必须接收 `InvocationHandler` 参数 |
| **实例化方式** | 直接 `new` | 只能通过 `Proxy.newProxyInstance` 工厂方法创建 |

**最重要的差别原因**：Java 是单继承语言。因为代理类必须继承 `Proxy`，所以它**无法再继承任何其他业务父类**，只能通过**实现接口**来定义行为。这就是为什么 JDK 动态代理**必须基于接口（`Star`）**，而不能基于类（`BigStar`）。

### 问题 2：我为什么要使用代理？代理的用途是什么？

**一句话核心答案**：**在不修改原有类（`BigStar`）源代码的前提下，给现有方法增加额外的功能（增强）。**

用你代码中的“明星”和“经纪人”来类比：

- 明星（`BigStar`）只管**唱**和**跳**（核心业务）。
- 经纪人（代理对象）负责**订场地、谈价格、处理琐事**（非业务功能）。

如果没有代理，你只能修改 `BigStar` 的 `sing` 方法，把订场地的代码硬塞进去，这会导致：

1. 明星类越来越臃肿（违反**单一职责原则**）。
2. 如果所有明星都要订场地，你得改所有明星类（代码重复）。
3. 如果哪天不想订场地了，又要改回去（维护困难）。

**代理在实际开发中的七大经典用途（即 AOP——面向切面编程）：**

| 用途 | 说明 | 现实场景（结合你的代码） |
| --- | --- | --- |
| **1. 日志记录** | 记录谁调用了什么方法、参数是什么 | `System.out.println("张三在" + 时间 + "唱了歌")` |
| **2. 性能监控** | 统计方法执行耗时 | 在 `invoke` 前后加 `System.currentTimeMillis()` |
| **3. 事务管理** | 数据库操作前后开启/提交/回滚事务 | 增删改查前开启事务，成功后提交，失败后回滚 |
| **4. 权限控制** | 检查当前用户是否有权执行该方法 | `if(用户不是VIP) { return "无权限"; }` |
| **5. 缓存控制** | 第一次查数据库存缓存，第二次直接取缓存 | Spring 的 `@Cacheable` 注解底层就是动态代理 |
| **6. 异常封装** | 将抛出的具体异常统一转为业务异常 | 将 `SQLException` 转为 `ServiceException` |
| **7. 远程调用（RPC）** | 本地调用方法，实际通过网络发给远程服务器执行 | Dubbo、OpenFeign 的底层核心原理 |

### 总结一句话

- **代理对象的类**是 JVM 在运行时临时造的，它披着 `Star` 的外衣（接口），但骨子里全是 `InvocationHandler`（经纪人）。
- **使用代理**就是为了**解耦**：让明星（业务逻辑）专心搞艺术，让经纪人（通用增强逻辑）专心搞杂务。这就是现代 Java 框架（Spring、MyBatis）赖以生存的基石。
